# own-repos-curator

A TUI for editing and managing introductory descriptions for each repository, centered around a list of your repositories. Written in Rust.

# Status
- Currently dogfooding.

# Use Cases
- For when you want to efficiently create repository descriptions.
  - *Target: Your public repositories (excluding forks and archived repositories)*

# Features
- Visualization through grouping and tagging.
- Various others (might be covered in the explanation below).

# Installation

Rust is required.

```
cargo install --force --git https://github.com/cat2151/own-repos-curator
```

# Execution

```
own-repos-curator
```

# Update

```
own-repos-curator update
```

# Explanation Generated by Codex CLI:

## What is this?

`own-repos-curator` is a TUI application designed to help you cultivate introductory descriptions for each of your public GitHub repositories, starting from your repository list.
It builds upon factual information obtained from GitHub, allowing you to organize a `1-line description`, `3-line description`, `tag`, and `group` locally, and save them as `repos.json`.

It's designed not just as a tool for viewing a list, but as a workspace for "preparing your public repositories so they can be easily introduced to others later."

## When is it useful?

- When your public repositories have grown, and it's hard to get an overview of what you've created yourself.
- When GitHub's description isn't enough to convey the full story, and you want to organize the purpose and highlights separately.
- When you want to gather source material first, as a preliminary step to creating descriptions for READMEs, portfolios, GitHub Pages, or dedicated showcase pages.
- When you were manually transcribing to spreadsheets or notes, but missed repository additions or updates became a painful task.
- When you want to quickly organize descriptions for multiple repositories using a keyboard-centric approach, without switching back and forth between browser tabs.
- When you want to organize "which repository to showcase in which category" using tags and groups.

It's particularly suited for those who continuously cultivate their own public repositories.
It helps reduce the situation where things you've built get buried because their descriptions haven't kept up.

## What makes it unique compared to existing solutions?

### 1. Fills gaps where GitHub's repository list and description fall short

GitHub offers mechanisms for listing repositories and editing descriptions, but it's not optimized for the experience of cross-repository description cultivation.
`own-repos-curator` synchronizes repository names, update times, and existing descriptions from GitHub, and allows you to manage "your own descriptive data" on top of that.

In short, its uniqueness lies not in replacing GitHub, but in using GitHub as a source of factual data while maintaining the descriptive context in a separate layer.

### 2. More specialized for repository management than spreadsheets or Notion

While repository descriptions can be managed with spreadsheets or Notion, repository addition and update information needs to be brought in manually.
This application allows you to perform repository list synchronization, description editing, tagging, group assignment, and filtering all within the same screen.

Its strength lies not in the flexibility of a general-purpose tool, but in its singular focus on "keeping public repositories in a presentable state."

### 3. Not just text editing, but also leaves reusable JSON assets

Descriptive data is saved in `repos.json`.
This makes it easy to repurpose for GitHub Pages, static sites, other tools, or generation scripts in the future, preventing data from being locked within the application.

Furthermore, this repository allows you to copy the JSON to related local repositories, automatically push to a backup repository on startup if configured, or manually commit/push from the TUI with `Shift+P`. This means it's not just "create a description and done," but rather structured to easily cultivate it as a public asset.

### 4. A TUI designed for a large volume of small edits

This application is better suited for quickly adding short descriptions and classifications to numerous repositories, rather than deep editing of a single repository.
Since tag assignment, group assignment, filtering, and sorting are all keyboard-centric, it's well-suited for a workflow of "gradually filling in descriptions."

## Past Challenges and How This Application Solves Them

### Challenge 1: Repository list management quickly becomes manual

Manually transcribing to introductory notes or lists every time a new repository is created is tedious.
As a result, there's a discrepancy between the repositories you want to showcase and those that are actually organized.

This application synchronizes the repository list from GitHub, so you no longer have to manually maintain the list.
The first solution is that your "inventory starting point" is automatically aligned.

### Challenge 2: GitHub descriptions alone provide insufficient introductory information

GitHub descriptions are short and make it difficult to include details like purpose, background, key highlights, or related categories.
On the other hand, writing everything in the README makes it difficult to compare or review them across a list.

This application allows you to retain the GitHub description while also having separate `1-line descriptions` and `3-line descriptions`.
This separation of "minimal GitHub explanation" and "supplementary introductory explanation" makes it easier to organize how you present your repositories.

### Challenge 3: As repositories grow, they become hard to find without classification axes

As the number increases, it becomes difficult to utilize them effectively without organizing axes like "Is it a CLI?", "Is it a Web app?", "Is it for learning?", or "Is it public-facing?".

This application allows for tags and groups, so you can organize repositories into meaningful clusters.
Furthermore, being designed with filtering and pagination in mind, it transforms a mere list into a searchable one.

### Challenge 4: Introductory data is scattered and hard to reuse

When descriptions are scattered across READMEs, notes, spreadsheets, and local JSON files, it becomes difficult to determine which is the latest version.
Additionally, if you want to repurpose them for a different public channel, formatting costs are incurred every time.

This application centralizes introductory information into `repos.json`, allowing you to unify the starting point for reuse.
This makes it easier to connect to future site generation or data publication, and also to incorporate into backup operations.

### Challenge 5: Organizing descriptions tends to be postponed

The task of opening each repository in a browser and writing descriptions is cumbersome, leading to descriptions often being postponed.

This application is a TUI well-suited for "quickly refining short descriptions while viewing a list," thereby reducing the cost of organizing descriptions.
As a result, it becomes easier to keep descriptions up-to-date after creating a repository.

## Summary

`own-repos-curator` is an application designed to transform your public repositories on GitHub from a mere "list of what exists" into a "list ready to be introduced to others."

Its uniqueness lies in integrating GitHub synchronization, description editing, tagging, grouping, and JSON asset creation into a single workflow, rather than separating them.
The role of this application is to streamline the previously scattered, manual management of repository introductions into a sustainable process.
